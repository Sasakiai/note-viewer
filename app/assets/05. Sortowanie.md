## Stabilność sortowania

Algorytm sortowania jest stabilny, jeśli zachowuje względną kolejność elementów o tej samej wartości.

Przykład:

- `Wejście:  4, 2a, 3, 2b, 1, 2c`
- `Wynik:    1, 2a, 2b, 2c, 3, 4`

Elementy `2a, 2b, 2c` zostają w tej samej kolejności.

Stabilność jest ważna np. przy sortowaniu rekordów po kilku atrybutach.

## SelectionSort

Idea:
1. znajdź minimum
2. zamień je z pierwszym elementem
3. powtarzaj dla reszty tablicy

Złożoność:

- `W(n) = Θ(n^2)`
- `A(n) = Θ(n^2)`
- `S(n) = O(1)`

Uwagi:
- działa podobnie długo dla prawie każdych danych
- standardowa wersja nie jest stabilna
- łatwy do zrozumienia, ale praktycznie zwykle gorszy od InsertionSort

## InsertionSort

Idea:
1. utrzymujemy posortowany prefiks tablicy
2. bierzemy kolejny element
3. wstawiamy go w odpowiednie miejsce w prefiksie

Złożoność:

- `W(n) = Θ(n^2)`
- `A(n) = Θ(n^2)`
- `S(n) = O(1)`

Uwagi:
- najlepszy przypadek: dane już prawie posortowane, wtedy czas może spaść do `Θ(n)`
- standardowa wersja jest stabilna
- na małych danych często jest lepszy niż bardziej złożone algorytmy

## Merge (funkcja)

Funkcja `merge` łączy dwa posortowane ciągi w jeden posortowany ciąg.

Przebieg:
1. porównujemy pierwsze niezużyte elementy obu ciągów
2. mniejszy przepisujemy do wyniku
3. przesuwamy wskaźnik w tym ciągu, z którego wzięliśmy element
4. gdy jeden ciąg się skończy, dopisujemy resztę drugiego

Jeśli przy równości bierzemy najpierw element z lewego ciągu, `merge` zachowuje stabilność.

Złożoność:

- `W(n) = Θ(n)`
- `A(n) = Θ(n)`
- `S(n) = Θ(n)`

## MergeSort

Idea:
1. podziel ciąg na dwie połowy
2. posortuj rekurencyjnie każdą połowę
3. połącz posortowane połowy funkcją `merge`

Pseudokod:

```
mergeSort(S, len) {
	if (len <= 1) {
		return S
	}
	
	m = len / 2
	
	left = mergeSort(S[0:m], m)
	right = mergeSort(S[m:len], len - m)
	
	return merge(left, m, right, len - m)
}
```

Złożoność:

- `W(n) = Θ(n log n)`
- `A(n) = Θ(n log n)`
- `S(n) = Θ(n)`

Uwagi:
- działa dobrze niezależnie od danych wejściowych
- jest stabilny, jeśli `merge` jest stabilne
- bardzo dobrze pasuje do list dowiązaniowych, bo scalanie można robić przez przepinanie wskaźników

## QuickSort

Idea:
1. wybierz pivot
2. funkcja `partition` ustawia elementy mniejsze lub równe po lewej, większe lub równe po prawej
3. rekurencyjnie sortuj obie części

Złożoność:

- `A(n) = Θ(n log n)`
- `W(n) = Θ(n^2)`
- `S(n) = O(log n)`

## Partition

Funkcja `partition` dzieli ciąg względem pivota:
1. wybieramy pivot
2. elementy mniejsze lub równe przesuwamy na lewo
3. elementy większe przesuwamy na prawo
4. po zakończeniu pivot trafia na swoje docelowe miejsce

Po `partition` pivot jest już ustawiony tak, jak będzie w posortowanej tablicy.

Uwagi o QuickSort:
- zwykle bardzo szybki na tablicach
- standardowa wersja nie jest stabilna
- najgorszy przypadek pojawia się przy bardzo złym wyborze pivota, np. gdy pivot ciągle dzieli dane skrajnie nierówno
- na listach dowiązaniowych zwykle wypada gorzej niż MergeSort

## CountSort

Założenia:
- sortujemy liczby naturalne,
- znamy maksymalną wartość `m`,
- `n` to długość ciągu.

Pseudokod:

```
countSort(a, l) {
	max = maxValue(a, l)
	l1 = max + 1
	
	counts[l1]
	result[l]
	
	for (i = 0; i < l1; i++) {
		counts[i] = 0
	}
	
	for (i = 0; i < l; i++) {
		counts[a[i]]++
	}
	
	for (i = 1; i < l1; i++) {
		counts[i] += counts[i-1]
	}
	
	for (i = l - 1; i >= 0; i--) {
		result[--counts[a[i]]] = a[i]
	}
}
```

`counts` po pierwszym przejściu:

`counts[x] = liczba wystąpień wartości x`

`counts` po sumowaniu prefiksowym:

`counts[x] = liczba elementów <= x`

Po ostatniej pętli:

`result = posortowany ciąg wejściowy`

Złożoność:

- `W(n, m) = Θ(n + m)`
- `A(n, m) = Θ(n + m)`
- `S(n, m) = Θ(n + m)`

Uwagi:
- nie jest sortowaniem przez porównania
- działa dobrze, gdy klucze są liczbami naturalnymi z niezbyt dużego zakresu
- w pokazanej wersji jest stabilny, bo ostatnia pętla idzie od końca wejścia

## RadixSort

Idea:
1. sortujemy liczby po kolejnych cyfrach
2. zwykle od najmniej znaczącej cyfry
3. do sortowania po cyfrze używamy stabilnego sortowania, np. CountSort

Uwagi:
- działa dobrze, gdy liczba cyfr jest mała
- nie jest sortowaniem przez porównania
- poprawność zależy od stabilności sortowania używanego na kolejnych cyfrach

## Stabilność omawianych sortowań

- SelectionSort - zwykle nie, bo zamiana minimum z początkiem nieposortowanej części może przestawić równe elementy
- InsertionSort - tak, bo element przesuwamy tylko przez elementy większe, więc równe nie zmieniają wzajemnej kolejności
- MergeSort - tak, jeśli w `merge` przy remisie bierzemy najpierw z lewego ciągu, więc równe elementy zostają w tej samej kolejności
- QuickSort - zwykle nie, bo `partition` robi zamiany, które mogą przestawić równe elementy
- CountSort - tak, w stabilnej wersji, bo wpisywanie do `result` od końca wejścia zachowuje kolejność elementów równych
- RadixSort - tak, jeśli sortowanie po każdej cyfrze jest stabilne; inaczej zepsuje się porządek ustalony wcześniej

## Dolne ograniczenie dla sortowań przez porównania

Każdy algorytm sortowania oparty tylko na porównaniach ma w pesymistycznym przypadku ograniczenie dolne:

`Ω(n log n)`

Dlatego CountSort i RadixSort mogą być szybsze, ale tylko dlatego, że nie opierają się wyłącznie na porównaniach elementów.

## Co porównywać na egzaminie

- czy algorytm jest stabilny
- czy wymaga dodatkowej pamięci
- czy działa dobrze na tablicy czy na liście
- jaki ma najlepszy, przeciętny i najgorszy przypadek
- czy jest sortowaniem przez porównania
