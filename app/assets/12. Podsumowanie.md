#PJATK #ASD

## Terminy

- specyfikacja algorytmu = nazwa, argumenty, warunek początkowy, warunek końcowy
- własność stopu = algorytm kończy działanie po skończonej liczbie kroków dla każdych poprawnych danych
- częściowa poprawność = jeśli algorytm się zatrzyma, to zwróci poprawny wynik
- całkowita poprawność = własność stopu + częściowa poprawność
- niezmiennik pętli = warunek prawdziwy przed iteracją i po iteracji
- operacja dominująca = operacja, której liczba wykonań najlepiej opisuje czas działania algorytmu
- rozmiar danych = parametr lub parametry opisujące wielkość wejścia
- częściowy porządek = relacja zwrotna, antysymetryczna i przechodnia
- ADS opisuje zestaw operacji, konkretna struktura danych opisuje sposób przechowywania

## Notacje asymptotyczne

- `O(g(n))` - górne ograniczenie
- `o(g(n))` - ścisłe górne ograniczenie
- `Θ(g(n))` - ten sam rząd wzrostu
- `Ω(g(n))` - dolne ograniczenie
- `ω(g(n))` - ścisłe dolne ograniczenie

Przykład:
- `n^2 + 5n + 2 = Θ(n^2)`
- `n^2 + 5n + 2` nie jest `O(n)`

Hierarchia, od wolniejszych do szybszych:
- `O(1)`
- `O(log n)`
- `O(√n)`
- `O(n)`
- `O(n log n)`
- `O(n^2)`
- `O(n^3)`
- `O(2^n)`
- `O(n!)`

## Złożoności

Wyszukiwanie:
- sekwencyjne: `W(n)=Θ(n)`, `A(n)=Θ(n)`, `S(n)=O(1)`
- skoki-co-k: `Θ(n/k + k)`, najlepiej dla `k = √n`
- binarne: `W(n)=Θ(log n)`, `A(n)=Θ(log n)`, `S(n)=O(1)`

Sortowanie:
- SelectionSort: `Θ(n^2)`
- InsertionSort: `Θ(n^2)`, a dla prawie posortowanych danych może zejść do `Θ(n)`
- merge: `Θ(n)` czasu, `Θ(n)` pamięci
- MergeSort: `Θ(n log n)` czasu, `Θ(n)` pamięci
- QuickSort: średnio `Θ(n log n)`, pesymistycznie `Θ(n^2)`
- CountSort: `Θ(n + m)` czasu i pamięci

Struktury danych:
- stos: `push`, `pop`, `top` w `O(1)`
- kolejka: `enqueue`, `dequeue`, `front` w `O(1)`
- deque: operacje na obu końcach w `O(1)`
- kopiec: `findMin O(1)`, `insert O(log n)`, `delMin O(log n)`
- HashMap: średnio `O(1)`, pesymistycznie `O(n)`
- BST: `O(h)`, dobrze zbalansowane `O(log n)`
- AVL: `search`, `insert`, `delete` w `O(log n)`

## Przebiegi

### `binarySearch`

1. bierzemy środek przedziału
2. jeśli trafiliśmy, zwracamy wynik
3. jeśli `key` jest mniejszy, idziemy w lewo
4. jeśli większy, idziemy w prawo
5. kończymy, gdy przedział zrobi się pusty

Warunki:
- dane muszą być posortowane
- potrzebny jest szybki dostęp do środka, więc tablica pasuje, lista nie

### `merge`

1. mamy dwa posortowane ciągi
2. porównujemy ich pierwsze niezużyte elementy
3. mniejszy przepisujemy do wyniku
4. po skończeniu jednego ciągu dopisujemy resztę drugiego

Jeśli przy remisie bierzemy z lewego ciągu, `merge` jest stabilne.

### `partition`

1. wybieramy pivot
2. elementy `<= pivot` ustawiamy po lewej
3. elementy `> pivot` ustawiamy po prawej
4. pivot trafia na swoje docelowe miejsce

Po `partition` pivot jest już ustawiony tak, jak w wyniku końcowym.

### `delMin`

1. minimum jest w korzeniu
2. zapamiętujemy korzeń jako wynik
3. ostatni element przenosimy do korzenia
4. usuwamy ostatnią pozycję
5. wykonujemy `downheap`, zamieniając z mniejszym dzieckiem
6. kończymy, gdy własność kopca wróci

### `upheap`

1. nowy element trafia na koniec kopca
2. porównujemy go z rodzicem
3. jeśli jest mniejszy, zamieniamy je miejscami
4. powtarzamy, aż dojdziemy do korzenia albo własność kopca wróci

### `downheap`

1. porównujemy element z jego dziećmi
2. jeśli łamie własność kopca, zamieniamy go z mniejszym dzieckiem
3. schodzimy poziom niżej
4. kończymy, gdy element jest już nie większy od dzieci albo gdy dojdziemy do liścia

### `CountSort`

1. zerujemy tablicę `counts`
2. zliczamy wystąpienia wartości
3. robimy sumy prefiksowe
4. przechodzimy od końca wejścia i wpisujemy elementy do `result`

Zapamiętaj:
- po zliczaniu: `counts[x] = liczba wystąpień x`
- po prefiksach: `counts[x] = liczba elementów <= x`
- na końcu: `result` jest posortowany

## Wzorzec pseudokodu na listę jednokierunkową

```
sumList(head) {
	sum = 0
	node = head

	while (node != null) {
		sum += node.value
		node = node.next
	}

	return sum
}
```

Własność stopu:
- w każdej iteracji przechodzimy do następnego węzła
- lista jest skończona, więc dojdziemy do `null`

## Stabilność sortowania

Stabilność oznacza, że elementy równe zachowują wzajemną kolejność.

Najczęściej:
- stabilne: InsertionSort, MergeSort, CountSort, RadixSort
- niestabilne: SelectionSort, QuickSort

Dlaczego:
- InsertionSort nie przestawia równych elementów względem siebie
- MergeSort jest stabilny, jeśli `merge` przy remisie bierze najpierw z lewego ciągu
- CountSort jest stabilny, gdy wpisujemy wynik od końca wejścia
- QuickSort i SelectionSort zwykle robią zamiany, które mogą przestawić równe elementy

## Sortowania: co z czym porównać

- MergeSort vs QuickSort:
  - MergeSort ma gwarantowane `Θ(n log n)` i jest stabilny
  - QuickSort zwykle działa bardzo szybko, ale ma pesymistycznie `Θ(n^2)` i zwykle nie jest stabilny
- CountSort/RadixSort vs sortowania przez porównania:
  - mogą być szybsze niż `n log n`
  - ale działają tylko dla bardziej szczególnych danych

Dolne ograniczenie dla sortowań przez porównania:

`Ω(n log n)`

## Jak dobierać strukturę do treści zadania

Myśl skrótem:
- czas dodania -> stos albo kolejka
- priorytet -> kopiec
- klucz -> HashMap
- porządek kluczy -> BST albo AVL
- szybki dostęp po indeksie -> tablica

Sygnał z treści zadania -> struktura:
- "ostatnio dodany", "najnowszy", "cofnij ostatnią operację" -> stos
- "najdawniej dodany", "pierwszy w kolejce", "obsługa w kolejności zgłoszeń" -> kolejka
- "najmniejszy" albo "największy priorytet" -> kolejka priorytetowa, zwykle kopiec binarny
- "szybko znaleźć po id/kluczu" -> HashMap
- "minimum, maksimum, poprzednik, następnik, przedział kluczy" -> BST albo AVL
- "dodawanie i usuwanie z obu końców" -> deque
- "często przechodzimy po elementach i wstawiamy po znanym miejscu" -> lista

Mini przykłady:
- klient chce podejrzeć ostatnio dodane zgłoszenie -> stos
- klient chce podejrzeć najdawniej dodane zgłoszenie -> kolejka
- klient chce zawsze obsłużyć zgłoszenie o najwyższym priorytecie -> kopiec
- klient chce szybko znaleźć klienta po `id` -> HashMap
- klient chce szybko znaleźć wszystkie klucze z przedziału `a..b` -> AVL albo BST

Uwaga:
- najdawniej dodany != najmniejszy priorytet
- najdawniej dodany sugeruje kolejkę
- najmniejszy lub największy według priorytetu sugeruje kopiec

## Mini FAQ

Jak uzasadnić własność stopu?
- wskaż zmienną sterującą i pokaż, że zbliża się do końca i nie może zmieniać się w nieskończoność

Kiedy HashMap jest zła?
- gdy potrzebujesz porządku kluczy, minimum, maksimum, poprzednika, następnika albo wyszukiwania zakresu

Kiedy BST jest gorsze od AVL?
- gdy dane mogą ułożyć się źle i drzewo może zdegenerować się do listy

Kiedy binarne nie ma sensu?
- na liście dowiązaniowej, bo środek nie jest dostępny w `O(1)`

Po czym poznać, że CountSort ma sens?
- gdy dane są liczbami naturalnymi i zakres wartości nie jest za duży
